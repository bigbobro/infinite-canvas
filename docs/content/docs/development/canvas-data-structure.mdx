# 画布数据结构

本文档说明当前画布在前端本地保存的数据结构、图片文件的存储和清理方式。

## 当前存储位置

当前画布项目主要保存在浏览器本地：

- 画布项目 JSON：`localForage`，数据库名 `infinite-canvas`，storeName `app_state`，key 为 `infinite-canvas:canvas_store`。
- 我的素材 JSON：`localForage`，数据库名 `infinite-canvas`，storeName `app_state`，key 为 `infinite-canvas:asset_store`。
- 图片 Blob：单独存到 `localForage` 实例，数据库名 `infinite-canvas`，storeName `image_files`。
- 视频等媒体 Blob：单独存到 `localForage` 实例，数据库名 `infinite-canvas`，storeName `media_files`。

画布 JSON 不直接长期保存大体积 base64 图片或视频。图片节点、视频节点、助手图片和素材媒体只保存展示 URL、`storageKey` 和元信息，真实 Blob 通过 `storageKey` 读取。

## 画布项目结构

每个画布项目是一个 `CanvasProject`：

```ts
type CanvasProject = {
  id: string;
  title: string;
  createdAt: string;
  updatedAt: string;
  nodes: CanvasNodeData[];
  connections: CanvasConnection[];
  chatSessions: CanvasAssistantSession[];
  activeChatId: string | null;
  backgroundMode: "lines" | "dots" | "blank";
  viewport: { x: number; y: number; k: number };
  ppt?: CanvasProjectPpt;
};
```

字段说明：

- `id`：画布项目 ID，当前前端生成。
- `title`：画布名称。
- `createdAt` / `updatedAt`：ISO 字符串。
- `nodes`：画布节点列表。
- `connections`：节点连线列表。
- `chatSessions`：右侧画布助手会话。
- `activeChatId`：当前选中的助手会话 ID。
- `backgroundMode`：画布背景模式。
- `viewport`：视口变换，`x/y` 是屏幕平移，`k` 是缩放比例。
- `ppt`：PPT 工作台工程的页面、方案和确认状态；普通画布不存在该字段。

PPT 页面与方案使用稳定身份，不使用排序或节点引用代替业务 ID：

```ts
type CanvasProjectPptPage = {
  pageId: string;
  index: number;
  confirmedNodeId?: string;
  takes: Array<{
    takeId: string;
    anchorNodeId: string;
    configNodeId: string;
  }>;
};
```

PPT 工程以 `compilePolicy` 区分两类互不兜底的 canonical 内容；页面账本只保存身份、页序、方案和最终确认，不重复保存标题或正文：

```ts
type CanvasProjectPptStyleContract = {
  source:
    | {
        kind: "preset";
        presetId: "clean-report" | "visual-story" | "brand-led";
      }
    | { kind: "custom" };
  direction: string;
  references: Array<{ storageKey: string }>;
};

type CanvasProjectPptDeckBrief = {
  version: number;
  sourceHash: string;
  audience: string;
  goal: string;
  narrative: string;
  styleContract: CanvasProjectPptStyleContract;
  globalRules: string[];
  forbiddenRules: string[];
  lockedDeckFacts: CanvasProjectPptLockedFact[];
};

type CanvasProjectPptSourceRef = {
  id: string;
  source: "material" | "requirements" | "user_answer" | "confirmed_assumption";
  excerpt: string;
  startLine?: number;
  endLine?: number;
  gapId?: string;
};

type CanvasProjectPptContentBlock = {
  id: string;
  kind:
    | "title"
    | "primary_claim"
    | "supporting_claim"
    | "body"
    | "list"
    | "table"
    | "chart_data"
    | "placeholder";
  text: string;
  sourceRefIds: string[];
  gapId?: string;
};

type PptContentState =
  | { status: "blocked"; gapIds: string[] }
  | { status: "reviewable" }
  | { status: "approved"; approvedAt: string };

type CanvasProjectPptPageSpec = {
  pageId: string;
  version: number;
  purpose: string;
  contentForm:
    | "cover"
    | "comparison"
    | "architecture"
    | "process"
    | "timeline"
    | "data"
    | "narrative"
    | "closing";
  contentFormNote?: string;
  sourceRefs: CanvasProjectPptSourceRef[];
  contentBlocks: CanvasProjectPptContentBlock[];
  contentState: PptContentState;
  lockedFacts: CanvasProjectPptLockedFact[];
  layoutRole:
    | "cover"
    | "section"
    | "content"
    | "evidence"
    | "comparison"
    | "close";
  layoutIntent: string[];
  visualEncoding: CanvasProjectPptVisualEncoding[];
  assetRefs: string[];
  freedom: string;
};

type CanvasProjectPptVerbatimSpec = {
  pageId: string;
  version: number;
  title: string;
  exactText: string;
  origin:
    | {
        kind: "source_slice";
        sourceHash: string;
        startLine: number;
        endLine: number;
      }
    | { kind: "user_edited" };
};

type CanvasProjectPptBase = {
  sourceMaterial: string;
  requirements: string;
  pages: CanvasProjectPptPage[];
  compilationSnapshots: CanvasProjectPptCompilationSnapshot[];
  anchorConfirmed?: boolean;
  skipAnchor?: boolean;
};

type CanvasProjectPpt = CanvasProjectPptBase &
  (
    | {
        compilePolicy: "structured";
        deckBrief: CanvasProjectPptDeckBrief;
        pageSpecs: CanvasProjectPptPageSpec[];
      }
    | {
        compilePolicy: "verbatim";
        verbatimSpecs: CanvasProjectPptVerbatimSpec[];
        confirmedGlobalSpec?: string;
      }
  );

type CanvasProjectPptCompilationSnapshot =
  CanvasProjectPptCompilationSnapshotBase &
    (
      | {
          compilePolicy: "structured";
          deckBriefVersion: number;
          pageSpecsVersion: number;
          deckBrief: CanvasProjectPptDeckBrief;
          pageSpecs: CanvasProjectPptPageSpec[];
        }
      | {
          compilePolicy: "verbatim";
          verbatimSpecs: CanvasProjectPptVerbatimSpec[];
          confirmedGlobalSpec?: string;
        }
    );
```

- `deckBrief.styleContract` 是唯一可编辑的 deck 级视觉事实源。`presetId` 只记录选择来源，真正参与 Compiler 的是当时冻结的 `direction`。
- `styleContract.references` 只保存本地图片 `storageKey`，不会为参考图伪造画布节点。图片清理会把这些 key 计入使用集合。
- structured 的标题、核心信息和其他正文都在 `contentBlocks`；每个非占位块必须引用能支持该文字的 `sourceRef`。`DeckBrief.sourceHash` 将整套受众、目标和叙事定位绑定到生成内容方案时的 material + requirements；页级来源则在 Builder、生成计划和 provider POST 前重新按行号切片并比对 excerpt。任一层漂移或 `contentState` 未到 `approved` 都会阻断生成。
- `PageSpec.layoutRole` 是页级职责，和只作用于当前方案的排版要求分开保存；`layoutIntent` 只能表达几何布局，`visualEncoding` 只能表达如何区分、强调、排序、分组或展示关系，两者都不能携带新的可见文案或事实。内容改写、占位或不呈现时，对应的 `lockedMapping` 和失效编码必须同步裁剪。
- verbatim 的 `exactText` 是唯一正文来源；原文切片必须带匹配的 source hash 和合法行号，用户明确编辑后改记 `user_edited`。`confirmedGlobalSpec` 只有经用户确认才逐字附加，不存在默认 Style Contract。
- Contract 真正变化会提升 DeckBrief 版本并清除全部页面确认；页面职责或布局变化只提升目标 PageSpec 版本并清除目标页确认。两者都保留方案、候选和 `compilationSnapshots`。
- 项目不读取旧 `mode`、页面 `title/outline/visualHint` 或节点文本作为 canonical 兼容路径；缺少 `compilePolicy` 的旧结构只能安全显示为待修复，不能进入生成或导出。

structured 新建向导在工程落盘前使用临时内容草稿承载缺口与审核：

```ts
type PptContentDraft = {
  revision: number;
  brief: PptContentBrief;
  pageSpecs: CanvasProjectPptPageSpec[];
  audit: {
    issues: PptContentAuditIssue[];
    gaps: PptInformationGap[];
  };
};

type PptInformationGap = {
  id: string;
  pageId?: string;
  kind:
    | "missing_detail"
    | "missing_evidence"
    | "unsupported_claim"
    | "ambiguous_input";
  question: string;
  reason: string;
  blocking: boolean;
  resolution?: PptInformationGapResolution;
};
```

材料或要求无法支持的内容会成为信息缺口，不会自动伪造成事实。用户回答和已确认假设会新增明确 sourceRef；省略或占位不能用于整套定位、标题和核心信息。编辑、删页、合并、排序、安全修复与单页重生成都绑定 `revision`，过期 preview/result 原子失败；只有不存在 blocking audit issue 时才能 finalize 为 `approved` PageSpec。

页面展示统一通过 `selectPptPageDescriptor(ppt, pageId)` 派生。selector 校验页面账本、连续页序、page/spec 数量与 `pageId` 双射，再读取 structured 的唯一 title/primary claim 或 verbatim 的 title/exactText；损坏时返回 `status: "invalid"`、标题“内容规格待修复”和具体原因。GenerationPlan 会在创建请求前排除 invalid 页面，导出会在读取图片 Blob 前失败。

## 节点结构

每个节点是一个 `CanvasNodeData`：

```ts
type CanvasNodeData = {
  id: string;
  type: "image" | "text" | "config" | "video";
  title: string;
  position: { x: number; y: number };
  width: number;
  height: number;
  metadata?: CanvasNodeMetadata;
};
```

通用字段：

- `id`：节点 ID。
- `type`：节点类型，当前有图片、文本、生成配置、视频四类。
- `title`：节点标题。
- `position`：画布世界坐标，不是屏幕坐标。
- `width` / `height`：画布世界坐标下的节点尺寸。
- `metadata`：节点内容和业务状态。

`metadata` 当前常用字段：

```ts
type CanvasNodeMetadata = {
  content?: string;
  prompt?: string;
  status?: "idle" | "success" | "loading" | "error";
  errorDetails?: string;
  fontSize?: number;
  generationMode?: "text" | "image" | "video";
  model?: string;
  size?: string;
  count?: number;
  naturalWidth?: number;
  naturalHeight?: number;
  freeResize?: boolean;
  isBatchRoot?: boolean;
  batchRootId?: string;
  batchChildIds?: string[];
  primaryImageId?: string;
  imageBatchExpanded?: boolean;
  inputOrder?: string[];
  storageKey?: string;
  mimeType?: string;
  bytes?: number;
  imageTask?: { taskId: string; model: string; expiresAt?: number };
  pptPageId?: string;
  pptTakeId?: string;
  pptGenerationRun?: PptGenerationRunSummary;
  pptGenerationRequest?: PptGenerationRequestTrace;
};
```

PPT 候选经“按要求改图”生成时，请求轨迹还会冻结下面的修改快照：

```ts
type PptCandidateEditSnapshot = {
  baseNodeId: string;
  globalInstruction: string;
  annotations: Array<{
    index: number;
    x: number;
    y: number;
    instruction: string;
  }>;
  finalPrompt: string;
};

// PptGenerationRequestTrace 中的可选字段
candidateEdit?: PptCandidateEditSnapshot;
```

不同节点的使用方式：

- 图片节点：`content` 是当前可展示的图片 URL，通常是 `blob:` URL；`storageKey` 指向本地图片 Blob；`naturalWidth/naturalHeight/bytes/mimeType` 保存原图信息。
- 视频节点：`content` 是当前可播放的视频 URL，通常是 `blob:` URL；`storageKey` 指向本地视频 Blob；`bytes/mimeType` 保存文件信息。
- 文本节点：`content` 保存文本内容；`fontSize` 保存字体大小；`prompt/status/errorDetails` 保存生成状态。
- 生成配置节点：`generationMode/model/size/count/inputOrder` 保存生成配置；`generationMode` 可选择文本、图片或视频；上游输入通过 `connections` 计算。
- 图片组节点：根节点用 `isBatchRoot/batchChildIds/primaryImageId/imageBatchExpanded` 记录批量生成结果；子图节点用 `batchRootId` 指回根节点。
- 异步生图任务：`imageTask` 是「猫佬（异步任务）」渠道的活动句柄，使页面刷新后能接着轮询而不是丢弃任务。明确完成、失败、取消或放弃后会清除；网络、取图或本地物化暂时失败且仍可按 task ID 取回时继续保留。`model` 存的是编码后的「渠道::模型」值，用于定位任务所属渠道。
- PPT 归属：`pptPageId` / `pptTakeId` 标记节点属于哪一页、哪一方案；`pptPageIndex` 只用于展示。
- PPT 生成轨迹：`pptGenerationRun` 记录一个页方案运行的摘要，`pptGenerationRequest` 记录一个实际付费请求槽的状态、远端 task ID、结果身份、不含 API Key 的冻结渠道身份与重复计费风险。count=1 时两者在同一输出节点；count>1 时 run summary 在 root，每个子节点保留自己的 request trace。`imageTask` 仅是活动任务句柄，不代替终态轨迹。
- PPT 修改追溯：修改 Request 本身已含 `requestId` / `runId` / `pageId` / `takeId`，`candidateEdit` 再记录基图、整页要求、点位序号与相对坐标、逐点说明和最终提示词。零点位直接使用原基图，有点位才临时生成带编号标记的请求图片；新结果只追加候选，页面的 `confirmedNodeId` 不因生成而改写。
- PPT 编译血缘：普通生成候选通过 `compilationSnapshotId` 指向不可变编译快照；修改稿只沿 `candidateEdit.baseNodeId` 递归找到根候选。确认和导出要求血缘唯一且全链同页、同方案；断链、循环、重复快照或跨页/方案借用都会先阻止操作，再读取图片 Blob。

## 连线结构

每条连线是一个 `CanvasConnection`：

```ts
type CanvasConnection = {
  id: string;
  fromNodeId: string;
  toNodeId: string;
};
```

连线只保存节点 ID，不保存端口坐标。渲染时根据节点位置和尺寸计算路径。

删除节点时会同步删除以该节点为起点或终点的连线。删除图片组根节点时，会把对应子节点一起删除。

## 助手会话结构

助手会话保存在画布项目内：

```ts
type CanvasAssistantSession = {
  id: string;
  title: string;
  messages: CanvasAssistantMessage[];
  createdAt: string;
  updatedAt: string;
};
```

消息结构：

```ts
type CanvasAssistantMessage = {
  id: string;
  role: "user" | "assistant";
  mode: "ask" | "image";
  text: string;
  isLoading?: boolean;
  references?: CanvasAssistantReference[];
  images?: CanvasAssistantImage[];
};
```

图片引用和助手生成图片也遵循同一套图片存储规则：

- `dataUrl` 字段当前可能是 `blob:` URL，也可能是旧数据中的 `data:image/...`。
- `storageKey` 存在时，以 `storageKey` 为准读取图片 Blob。
- 发送到 AI 接口前，如果接口需要 base64，会通过 `imageToDataUrl` 临时把 Blob URL 转成 data URL。

## 图片写入流程

所有新增图片应通过 `uploadImage(input)` 写入：

1. 传入 `Blob` 或 data URL。
2. 内部转成 `Blob`。
3. 生成 `storageKey`，格式为 `image:<id>`。
4. 把 Blob 写入 `image_files`。
5. 创建 `blob:` URL，并缓存在内存 `objectUrls`。
6. 读取图片宽高，返回：

```ts
type UploadedImage = {
  url: string;
  storageKey: string;
  width: number;
  height: number;
  bytes: number;
  mimeType: string;
};
```

图片节点会通过 `imageMetadata(image)` 写入：

```ts
{
  content: image.url,
  storageKey: image.storageKey,
  status: "success",
  naturalWidth: image.width,
  naturalHeight: image.height,
  bytes: image.bytes,
  mimeType: image.mimeType
}
```

因此，`content` 只适合当前浏览器会话展示，不能作为长期文件标识；长期标识是 `storageKey`。

## 图片读取和旧数据迁移

打开画布时会执行图片补水：

- 如果图片节点有 `storageKey`，通过 `resolveImageUrl(storageKey, fallback)` 读取 Blob 并生成新的 `blob:` URL。
- 如果图片节点没有 `storageKey`，但 `content` 是旧的 `data:image/...`，会调用 `uploadImage(content)` 迁移到 `image_files`，并补上 `storageKey`。
- 助手消息里的引用图和生成图也会执行同类逻辑。

我的素材读取时也会做迁移：

- 有 `storageKey`：恢复 `coverUrl` 和 `data.dataUrl` 的可展示 URL。
- 无 `storageKey` 且保存了 base64：写入 `image_files`，然后更新素材里的 `storageKey`。

## 图片移除和清理

图片不是在删除节点时立即按节点逐张删除，而是做引用清理：

1. 删除节点、清空画布、删除画布、删除素材、删除助手会话时，会触发 `cleanupImages`。
2. `cleanupImages` 会收集当前仍被画布项目、素材和额外传入数据引用的所有 `storageKey`。
3. `cleanupUnusedImages` 遍历 `image_files` 中的全部图片。
4. 不在引用集合里的图片会被删除。
5. 删除时会同时 `URL.revokeObjectURL`，并从内存缓存 `objectUrls` 移除。

这套方式可以避免同一张图片被画布、素材或助手同时引用时误删。
