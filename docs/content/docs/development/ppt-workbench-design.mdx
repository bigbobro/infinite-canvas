# PPT 工作台设计文档

> **文档性质**：本文档是 PPT 工作台模块的活文档（living doc），随功能迭代持续更新，反映的是当前实际状态，不是一次性设计快照。当前主流程为材料 → 可编辑大纲 → 视觉方向 → 建图落画布 → 面板批量生图（含首页锚定）→ 确认此页 → 下载图片 ZIP 或图片版 PPTX。在此基础上还支持「已有规格，直接生图」模式——用户已在外部把每页完整提示词写好，工具只做逐字展开，不改写。新会话/新 agent 可从本文档直接理解当前架构，具体代码坐标见附录（合并上游后以符号名重新定位行号）。

## 1. 背景与目标

在现有生图能力之上，新增「PPT 工作台」模块，支撑以下业务流程：

1. **输入原始材料**：用户提供文字材料（Markdown 或整份文档），明确 PPT 核心内容。
2. **生成分页大纲（节点化）**：LLM 基于材料生成分页大纲（如 1–9 页），每页在画布上对应一个独立节点。
3. **视觉方向（前置配置）**：生图前选择默认方向，或在高级区填写自定义方向、添加本地参考图；整套 deck 只保存一份 `DeckBrief.styleContract`。
4. **批量生图**：结合大纲与风格，每页分别生成 16:9 页面图。
5. **多轮精修**：对每页生成结果做第二轮、第三轮乃至多轮调整。
6. **批量下载**：全部页面确认后，可下载整套页面图片 ZIP 或图片版 PPTX。

## 2. 范围界定

- **当前交付物仍是整页图片**：最终检视可下载按页序命名的图片 ZIP（`01_封面.png` … `09_总结.png`），也可把同一组最终确认图按整页铺入 `.pptx`；图片版 PPT 内的文字不可单独编辑，不等同于原生对象级可编辑 PPT。
- **维持纯前端**：不引入任何后端/服务端持久化，数据全部走现有 localforage/IndexedDB 体系。
- **一个 deck = 一个专属画布工程**：PPT 结构数据住在 `CanvasProject` 内，导出/导入/WebDAV 同步免费获得，单一事实源。

## 3. 架构决策

### 3.1 选型：向导 + 画布 + 面板（混合架构）

对比过三条路：

- **方案 A：独立表单式工作台**（复刻 `/image` 三栏模式）——排除。步骤 2/5 与画布割裂，精修体验要在表单页重造一遍蒙版重绘/迭代链，重复建设。
- **方案 B：纯画布模板**——排除。没有流程引导、页状态追踪和批量下载落点，相比手动拼节点价值增量太小。
- **方案 C（采用）：混合架构**：
  - **步骤 1–3 用向导承载**：新路由 `/ppt`（进顶部导航，与生图/视频工作台并列），deck 列表 + 新建向导。向导第一步先选生成方式，两种模式共用后续落画布/面板/导出流程：
    - **「从材料拆大纲」**（默认）：材料输入 → LLM 生成分页大纲（**向导内可编辑：改标题/要点、增删页；确认后才落画布**）→ 选择视觉方向。适合还没想清楚具体每页内容、需要工具帮忙规划的场景。
    - **「已有规格，直接生图」**：适合用户已经在外部（如与 AI 多轮对话）把每页要发给生图模型的完整提示词都写好的场景。工具的职责从「理解与规划」收窄为「定位与搬运」——LLM 只输出每页在原文中的起止行号，正文由代码从原文按行逐字切片，改写在结构上不可能发生，不需要事后校验。未被页面切片占用的全局视觉说明会作为自定义方向草稿带到第三步，由用户确认；为空时仍默认使用“清晰专业”。
  - **确认后程序化建图**：构造完整 `CanvasProject`（节点 + 连线 + `ppt` 字段）经 `importProject` 入库，跳转 `/canvas/:id`。建图不需要画布页面在场。
  - **步骤 4–6 用 PPT 工作台承载**：页状态总览、批量生成、单页精修、确认此页与最终检视；结构画布作为高级视图保留。
  - **步骤 5 在工作台统一动作语义**：“再生成一稿”是无修改要求的单张探索，“按要求改图”支持整页要求与可选点位；蒙版重绘、裁剪、放大等画布原子能力继续保留。

### 3.2 执行层：PPT 私有的可靠生成模块

`project.tsx` 是上游活跃热区，PPT 仍只在这里挂载一个 project-local 模块，但付费生成不再经通用 `run_generation`。`createGenerationPlan()` 只产生可序列化的 `structureOps + pptOps + batch/runs/requests`；`PptGenerationModule.start(plan)` 统一执行结构写入、flush、从同一 localforage adapter 读回、持久化 `submitting` 锁，再放行实际 POST。

这样既保留通用 Agent/画布生成路径不变，又把 PPT 的付费门禁、固定请求槽、task 恢复、结果物化和终态通知收在一个深模块内。PPT 面板与单页工作区都只能调用同一个 `start(plan)`，不能自行 apply 结构后触发请求。当前完整边界以 §7.4 侵入点清单为准。

顺带为 P2 铺路：将来加 `workbench_ppt_*` Agent 站点工具（本地 Codex/Claude 一句话生成整套 deck）几乎免费，落在「MCP/Agent 操作画布」这条差异化护城河上。

### 3.3 视觉方向：DeckBrief Contract 是唯一事实源

`DeckBrief.styleContract` 是唯一可编辑的 deck 级视觉事实源，保存方向来源、冻结的方向文本和 `{ storageKey }[]` 参考图。预设 ID 只记录用户当时从哪个目录项开始，Compiler 永远消费已经写入 Contract 的 `direction`；未来调整预设目录不会静默改变旧 deck。

新建工程不创建风格文本或风格参考图节点。即使画布中出现旧的 `pptRole: "style"` 投影节点，Compiler 与 GenerationPlan 也必须忽略它。普通画布上游输入仍按原生连线语义参与当前方案，但不能覆盖 deck Contract。

### 3.4 prompt 冻结

通用画布的 `handleGenerateNode` 仍可能把组装后的 prompt 回写到 config 节点，但 PPT 已不再进入这条路径。生成计划创建时以当时的节点与连线调用 `buildNodeGenerationContext()`，把最终有效 prompt 和选中的图片引用 ID 冻结到 Request；执行模块只消费这份计划，不再从运行中可能变化的 `metadata.prompt` 二次取值，也不持久化 API Key、认证头或图片 base64。

### 3.5 按要求改图：整页要求 + 可选点位

图节点的统一修改入口支持三种等价组合：仅整页要求、仅一个或多个点位、整页要求 + 多点位。两类输入可独立清除，提交前编译成同一份冻结 `PptCandidateEditSnapshot`，修改清单、最终提示词、确认弹窗和真实请求都消费该快照。提示词优先级是点位要求 > 整页要求 > 其余内容保持不变。

零点位时直接使用原基图，最终提示词不包含红色标记、所指对象或不存在的局部区域；有点位时才生成带编号气泡的临时请求图片。PPT 候选的“生成修改稿”固定创建 1 个 image-to-image Request 并追加 1 个候选，不覆盖基图，不改写 `confirmedNodeId`。

不走蒙版路线：蒙版是「刷一片区域 + 一段描述」，一个 mask 只能表达一个区域集配一个 prompt，表达不了「①改成 X、②改成 Y」这种多点各改各的——mask 不携带身份。两者并存、各管各的活：蒙版擅长大面积区域重绘，annotate 擅长多个离散对象各改各的。

同一界面仍可用于普通画布图节点；普通画布保留原直连生成路径，PPT 候选则必须进入私有 Generation Module，记录 Run/Request 及修改快照后才发请求。

## 4. 数据模型

```ts
type CanvasProjectPptStyleContract = {
    source:
        | { kind: "preset"; presetId: "clean-report" | "visual-story" | "brand-led" }
        | { kind: "custom" };
    direction: string;                         // Compiler 消费的冻结文本
    references: Array<{ storageKey: string }>; // 本地 Blob 长期标识
};

type CanvasProjectPpt = {
    sourceMaterial: string;
    requirements: string;
    deckBrief: {
        version: number;
        audience: string;
        goal: string;
        narrative: string;
        styleContract: CanvasProjectPptStyleContract;
        globalRules: string[];
        forbiddenRules: string[];
        lockedDeckFacts: CanvasProjectPptLockedFact[];
    };
    pageSpecs: Array<{
        pageId: string;
        version: number;
        layoutRole: "cover" | "section" | "content" | "evidence" | "comparison" | "close";
        layoutIntent: string[];
        // 另有来源、锁定正文/事实、自由度和人工确认字段
    }>;
    pages: Array<{
        pageId: string;                       // 稳定页面身份
        index: number;                        // 页序（1 起）
        title: string;
        outline: string;
        visualHint: string;
        confirmedNodeId?: string;
        takes: Array<{ takeId: string; anchorNodeId: string; configNodeId: string }>;
    }>;
    compilationSnapshots: CanvasProjectPptCompilationSnapshot[];
    anchorConfirmed?: boolean;
    skipAnchor?: boolean;
    mode?: "outline" | "extract";
};

// CanvasNodeMetadata（types/canvas.ts）增加的可选标记字段
pptPageId?: string;
pptTakeId?: string;
pptPageIndex?: number;                                  // 只用于展示
pptRole?: "outline" | "style" | "page" | "source";   // "source" = 「已有规格，直接生图」模式下承载原始规格稿的节点
pptGenerationRun?: PptGenerationRunSummary;             // root 唯一持有
pptGenerationRequest?: PptGenerationRequestTrace;       // 每个固定请求槽唯一持有，终态保留
```

项目尚未上线，PPT 数据直接采用当前结构，不保留旧视觉字段迁移。`styleContract` 是唯一 deck 视觉事实源；`layoutRole` 是页级职责，方案排版仍由当前 take 的配置保存。`pageId`/`takeId` 是页面、方案、Run、Request、候选归属与通知定位的业务身份；`index` 只负责页序展示，anchor/config ID 只负责节点引用。

**防御性解析原则**：面板以 `ppt.pages` 为纲、以 metadata 标记做校验，发现引用或台账缺失时给出可复制诊断而不崩溃。普通位置、尺寸与视角编辑仍留在结构画布；Run/Request 所有者、生成归属连线和未解决方案的输入配置由工作台保护，不能通过普通删除、替换、Agent、撤销或图片派生工具绕过。方案删除统一走工作台，并在运行中或待处理时阻止删除。

`ppt.mode` 决定面板对该 deck 走哪套生成参数（详见 §3.4）；`pptRole: "source"` 节点只在「已有规格，直接生图」模式下出现，承载用户粘入的原始规格稿全文，**默认不与任何 config 节点连线**——它包含全部页面内容，连了会把全篇原文混进单页 prompt。画布原生支持用户自己手动拉线，工具不代为决定。

## 5. 画布工程结构（deck 的形状）

```
DeckBrief.styleContract ──→ Compiler ──→ 冻结 prompt / referenceSnapshots

[第1页大纲 文本节点]──→ [第1页 生成配置] → (生成后出现) [第1页 页面图] → 精修链向右生长…
（每页一列，横向排布；第 2–N 页结构同上）
```

生成计划在付费请求前预创建 root/request 节点及连线：count=1 时 root 与 request 是同一节点；count>1 时 root 持 Run 摘要，N 个 child 分别持 Request trace。成功结果在原 request 节点上物化为候选，不另建第二份结果节点。

**一页多线路（take）**：一页可以有多条平行线路，每条线路是一对独立的 outline/config 节点（图中「第1页大纲」+「第1页 生成配置」只是其中一条）。面板「新增线路」追加一条新线路，**不销毁旧的**——原 outline/config 节点、已确认图、锚定状态都保持不动，旧线路仍是合法候选。新线路的 outline 节点初始内容复制自当前显示的那条线路（用户要的是「改」，不是「重写」）。

**P0 当前摆位**：新节点不落在当前页那一行的正下方。deck 建图的行距与「下一行」的间距相等，落在本行下方必与下一页那一行重合；因此当前实现取**整个 PPT 网格（所有带 `pptPageIndex` 的节点）的全局最低点**再加一个行距新起一行，x 沿用最新一条线路的列，并把画布视角带到新节点。P1 不改变这套底层节点摆位，但会用 §11 的页级 Focus 取代这段全局画布定位交互。

**P0 当前候选挑选**：每页的候选图（每条线路 config 下游全部成功生成的图，剪枝规则见 §6）按线路分组、组内保持创建序，铺成一条缩略图条，**候选数 ≥2 才显示**；点缩略图跳画布视角并选中该节点，hover 可直接「用这版 / 取消确认」。2round 实测已确认该窄面板方案无法支持辨认和比较，P1 按 §11 改为 Focus 内的 16:9 大图候选。

## 6. 关键设计难题与对策

1. **视觉一致性（成败点）**——三层叠加：
   - 每页 prompt 都由同一份冻结 Contract direction + 页面职责 + 页面规格编译；
   - Contract 参考图以 `storageKey` 快照进入每个请求，并与普通上游图片去重；
   - **「首页锚定」两段式批量，默认开、可跳过**：先只生成第 1 页 → 用户精修确认 → 把确认图自动 `connect_nodes` 进第 2–N 页的配置节点 → 再批量生成。
2. **文字渲染准确性取决于模型能力**：当前使用的文字渲染强模型已经可以产出高信息密度的中文页面，不再把“中文小字天然不可用”作为产品前提。仍需按具体模型和页面实测，个别错字可用按要求改图或蒙版重绘修正。当前 ZIP 和图片版 PPTX 都以整页图片为内容，文字不能单独编辑；“图片型”描述的是交付格式，不再限制内容只能是低文字量、视觉冲击型页面。
3. **并发与失败处理**：现有机制无并发上限（生图工作台 count=15 也是全并发），9 页规模 MVP 直接沿用；**必须给单页失败重试**（面板每页独立重生成按钮），一页失败不毁整批。更长 deck 的并发队列（3–4 并发）留 P1。
4. **大纲质量**：向导内大纲**可编辑**（改标题/要点、增删页）后才落画布——大纲错了后面全错，这一步的人工把关是流程设计的一部分。
5. **候选归属先看稳定身份与台账，再做有边界的图遍历**：`pageId + takeId`、Run/Request trace 是生成结果的主归属；仅对尚无台账的结构结果，从该方案 config 向下做剪枝遍历，并在其他方案边界停止，避免首页锚定连线把整套 deck 吞进同一页。`count > 1` 时 N 个固定 Request child 各自就是候选，root 只保存 Run 汇总与当前主图投影；不再用页序或“只收 batch root”推断候选血缘。

## 7. 上游同步策略（fork 迭代管理，保守评估制）

本仓库是 `basketikun/infinite-canvas` 的二开 fork（remote 已配好 `origin` + `upstream`）。

### 7.1 原则：上游发版只触发「评估」，不自动触发「合并」

每次上游发版，先做影响评估（读 release notes + diff 侵入点清单与依赖面），然后显式决策：

- **情况 1：上游新功能对我们无影响** → 仍要主动决策「并还是不并」。默认**不并**；只有明确想要该功能/修复时才并。不追新。
- **情况 2：上游改动影响到我们**（改了侵入点文件、`applyAgentOps` 管线、生成服务层等依赖面）→ 评估「改还是不改」：跟进适配，或暂缓合并冻结在当前基线。

每次评估**留决策记录**（追加到下表），合并后执行固定核对流程。

| 位置 | 改了什么 | 为什么不能只做加法 |

## 10. 决策记录

**已确认（用户拍板，P0）**：

- 混合架构（向导 + 专属画布工程 + 画布内面板）；
- P0 当时只交付图片 ZIP；图片版 PPTX 后续作为独立高级导出能力实现，现已完成且仍不属于原生可编辑 PPT；

- 上游同步保守评估制（第 7 节）。

**已确认（用户拍板，「已有规格，直接生图」模式）**：

- **D1 本期范围**：只解决「用户已在外部把每页内容规划完整、写出完整生图提示词稿」这一类需求。「从零帮用户把模糊想法规划/改写成规格」的对话式扩展留待后续验证，不在本期做（见 D6）。
- **D2 改动范围的澄清**：不是「只能改向导」。向导第一步的模式选择只是**入口**改动；只要能让这个新生成方式正确工作，下游的建图逻辑（`deck-builder.ts`）、画布面板（`canvas-ppt-panel.tsx`）等文件都可以按需调整——§3.4 的 `prompt` 回写挡板正是靠面板改动解决的。
- **D4 模式选择方式**：不做「自动识别输入类型 + 告知判断结果 + 允许用户推翻」，改为向导上**显式**选择两种模式之一，默认落「从材料拆大纲」（老流程）。理由：两种误判代价不对称——把一份已经写好的完整规格稿误判成需要重新规划，会导致内容被复述改写，等于线上复现了本期要修的问题；而显式选择的实现更小、误判率为零。
- **D5 规格节点默认不连线**：用户粘入的原始规格稿在画布上单独建一个节点保存全文（单一事实源），但**默认不连**任何生成配置节点——它包含全篇内容，连了会把全篇原文混进单页 prompt。画布原生支持用户自己手动拉线，工具不代为决定。
- **D6 对话式扩充暂不做**：帮助只有模糊概念、还没规划好每页内容的用户通过多轮对话把内容「聊」出来，是有价值的方向，但作为独立能力**是否可靠尚未验证**，本期不做，留作后续任务。

**已确认（用户拍板，PPT 多版本模式）**：

- **D7 候选归属采用稳定身份守卫的剪枝 BFS**：结构关系仍由 config 下游遍历决定；新模块预建的输出节点同时写入 `pptPageId/pptTakeId`，遍历遇到其他页面或方案的稳定身份立即停止。`pptPageIndex` 只展示，不再承担业务归属。
- **D8 一页多线路用 `takes[]`，不做画布自动视觉分组**：归属靠 D7 的剪枝 BFS 保证正确，不需要额外的视觉分组机制兜底；画布原生 Group 是几何包含（拖进框即属于该组，拖出即脱离），生成的图落在算好的坐标上未必落进组框，不能拿来做归属判定。P1 仍不整理底层画布，但会按 §11 在页级 Focus 中聚合显示，不再要求用户靠拖节点完成精修定位。
- **新线路初始内容 = 复制原文**：新增线路时 outline 节点内容复制自当前线路，而不是留空或让 AI 改写——用户要的是「改」，不是「重写」，给一份副本直接编辑最省事。
- **候选排序 = 按线路分组 + 组内节点数组序，不引入 `metadata.seq`**：`CanvasNodeMetadata` 没有时间戳字段，「最新」原本就是「`project.nodes` 数组末位」这个隐含约定；铺开候选后按线路分组展示（这版是哪条线路出的一目了然）已经足够说清顺序，不必新增字段——数组序会被 undo/import 扰动的风险先接受，真出问题再加序号。

**已确认（用户拍板，标注改图 annotate）**：

- **技术路线选 B（烧标记单图），不选 A（坐标转文字）/ C（双图）/ 蒙版**：A 在密集目标（图标挨在一起）时纯文字描述指认不清，实测还丢了保真细节（圆环）；C 成本翻倍，且唯一优势（结构上防标记泄漏）经实测非必要——B 的标记从未泄漏；蒙版的 mask 不携带身份，表达不了「多点各改各的」（一个 mask 只能配一个区域集 + 一个 prompt）。7 次真实 API 调用全部验证过：仅给位置、不给物体名即可准确命中，不需要额外的视觉识别步骤。
- **挂载方案选 B：自持状态的宿主组件，不照抄蒙版对话框的挂载范式**：蒙版对话框在 `project.tsx` 里有 8 个有状态触点，是上游自己的代码风格；二开照抄等于把这个巨型组件从 2 触点炸到 9，「复用其挂载点」也不成立，等于深度侵入另一个上游文件。改为状态住二开自有 store（`useAnnotateStore`），节点数据经 `canvasContext.snapshot` 读，生成路径自走 `requestEdit` + `applyOps` 建子节点/写状态。该方案在 `project.tsx` 落下 4 处触点，全部为加法（无一改动上游既有行为或函数体），但超出本清单原定「1 import + 1 挂载」上限，已获用户批准破例（2026-07-15），条件是全部为加法、且每处带 `[二开]` 标记（§7.4 #5 已按此登记）。
- **否决工具面板版本常量 v6 → v7 升级**：曾考虑借上游「版本号递增」的既有惯例，让存过工具面板配置的浏览器自动看到新工具。否决理由：① 改的是上游既有行，且升版会重置用户已保存的工具自定义；② 当下收益为零——本机没有存量配置，annotate 按默认工具列表已经可见。已知边角记档不改码：存过 `canvas-image-quick-tools-v6` 配置的浏览器需要用户自己在工具面板里勾选，才能看到「标注改图」。

**留待实现期细化**：风格预设的具体文案清单、并发队列阈值、16:9 的具体 size 档位映射。

## 12. 当前迭代实施：工作台主界面化与方案分支提示词

P1 同步实测带来了一个比“整理画布”更根本的结论：对 PPT 工程而言，用户的主对象不是节点和连线，而是 `页面 → 方案分支 → 候选稿`。现有页级 Focus 已经比全局画布更接近真实工作方式，因此后续不应继续让 PPT 工作台附属于画布；**PPT 工作台应成为默认主界面，结构画布退为按需进入的高级视图**。

这里统一把当前界面的“线路”定义为**方案分支**。新建分支的业务含义是“保留当前方案，同时修改提示词探索另一种方案”，所以系统必须先复制当前分支提示词，给用户编辑空间，再由用户主动生成。分支尚无候选稿时可以继续编辑；一旦已有候选稿，原分支提示词保持稳定，后续调整通过“基于此方案调整”派生新分支，避免同一分支的候选实际来自不同提示词。页面的最终确认仍属于页面，指向任一分支中的一个候选稿。

架构上暂不直接建立独立 `/ppt/:id` 运行时。当前 `project.tsx` 和 `canvasContext` 仍承载节点状态、生成、取消、持久化与 MAOLAO 异步续跑；复制这套能力风险过高，在合并后的新上游结构确定前抽取运行时也会放大冲突。近期采用**同一运行时、两个界面**：PPT 工程默认显示覆盖画布内容区的全尺寸 Workbench Surface，画布保持挂载；用户需要理解节点结构或做高级编辑时切换到“结构画布”。非 PPT 工程不改变。

分期顺序如下：

1. **先补功能**：在现有页级工作区加入分支提示词草稿、编辑和“基于此方案调整”；创建分支与发起生成拆开，不自动消耗 API。
2. **再升主界面**：把 Modal 形态的 Focus 提升为 PPT 工程默认的全尺寸工作台，补工作台/结构画布切换和双向定位。
3. **远期再拆运行时**：完成当前实测和上游合并后，只有在独立路由或后台渲染确有收益时，才另评估共享 `ProjectRuntime`；不得为 PPT 复制生成和任务恢复逻辑。

阶段 1、2 已在 `ppt-workbench-primary-surface` 迭代中实现，并通过零 API 浏览器复核和用户体验验收：PPT 工程默认显示全尺寸工作台；新建方案分支先编辑提示词，创建与生成分离；已有候选的分支通过“基于此方案调整”派生；结构画布保持挂载，并可在切换时定位当前候选或分支节点。浏览器复核还覆盖了未保存提示词拦截、隐藏画布键盘隔离、标注大视图、最终检视、刷新持久化、跨工程状态，以及返回结构画布后的真实拖动、缩放、节点选中与点击。该迭代本身继续复用 `CanvasProject`、`canvasContext` 和原生成管线，没有抽取第二套运行时，也没有发起真实模型调用；MAOLAO 生成中刷新续跑随后已在上游合并任务按真实路径验收。

工作台继续约束“已有候选后提示词只读”。结构画布作为高级入口只负责位置、尺寸、视角与安全查看；PPT 的 outline、style、source、config、候选和 Run/Request 归属始终由工作台管理，不能经通用编辑、生成、重试、替换、删除、连接、撤销或 Agent 绕过。画布多线路自动分组、自动整理、永久重排和独立运行时继续后置。详细决策见 `CONTEXT.md`、`docs/adr/0001-ppt-workbench-primary-surface.md` 和 `.trellis/tasks/archive/2026-07/07-17-ppt-workbench-primary-surface/`。

## 13. 第三批收口：唯一读模型、生成计划与事务式编辑

第三批没有建立第二套 PPT 运行时，而是把「页面 → 方案 → 候选稿」的读、算、删、生成收敛到三个深模块：`page-workspace.ts` 负责方案所有权，`generation-plan.ts` 负责生成预演与执行，`CanvasPptPromptEditor` 负责本地编辑事务。UI 只消费结果，不重复遍历节点图或重算费用。

### 13.1 方案所有权与删除边界

`buildPptPageWorkspace(project, page)` 是唯一方案读模型。页面与方案分别以必填的 `pageId` / `takeId` 作为业务身份，`index` 只用于排序和展示，`configNodeId` 只是可替换的结构引用。从 config 向下遍历时，在其他 take 的 outline/config 边界与不同 `pptPageId` / `pptTakeId` 处截断，并按 take 顺序首次认领输出：

- `candidates`：本 take 成功且不是 batch root 的图片；
- `ownedOutputNodeIds`：成功、失败、loading 与 batch root 在内的全部认领输出；
- `generationRequests` / `unresolvedGeneration`：归属该方案的持久化请求轨迹，存在未解决请求时禁止把该页当作“未生成”重复批量下发；
- `deleteNodeIds`：全部认领输出加本 take 的 outline/config，直接交给共享删除命令；
- `resolvedConfirmedNodeId`：存量确认若指向 batch root，只读解析到其 `primaryImageId`，不写迁移。

PPT 方案元数据不进入普通画布历史，因此方案删除必须先从 `past/future/lastHistory` 裁掉完整删除集及关联连线，再停止相关生成并复用私有 `deleteNodes`。这样普通画布删除仍可撤销，而已删除方案不会被撤销复活为孤儿节点，也不会继续占用图片 Blob 引用。

### 13.2 `GenerationPlan` 是费用预演与执行的同一事务

```ts
createGenerationPlan(
  intent: GenerationIntent,
  { project, effectiveConfig },
): GenerationPlan
```

计划同时返回 `batchId`、`runs`、有序 `structureOps`、纯数据 `pptOps`、`pageCount`、`callCount`、`callBreakdown` 与 `excludedPages`。一次用户确认对应一个 batch，每个 `pageId + takeId` 对应一个 run，run 内按实际张数预先建立固定 request 槽。契约如下：

1. 弹窗、按钮费用文案与真正执行必须消费同一个 plan；禁止 UI 自己用“页数”估算调用数。
2. `callCount = Σ run.requests.length`；`callBreakdown` 按 `buildNodeGenerationContext` 实际收集到的参考图分成文生图/图生图，包含计划内尚未落盘的锚定连线。确认后把参考图快照转成独立 data URL，POST 不再按可变节点 ID 重新读取内容；`inputRefs` 继续用于 durable 引用校验和诊断。
3. 缺少 config 的页进入 `excludedPages`，不进入 `runs`、页数或费用；`deriveAndGenerate` 的新方案、节点和连线也必须在同一 plan 内。
4. `PptGenerationModule.start(plan)` 必须按 `draft → persisted → submitting` 分段落盘并从同一 localforage adapter 读回校验；只有全部请求槽的 durable `submitting` 锁都可读回后才允许 POST。
5. 模型、尺寸与张数统一经 `resolveGenerationConfig` / `getGenerationCount` 解析；摘要、配置面板、计划与真实请求不得各自维护 resolver。
6. `composerContent` 非空只决定进入配置合成器模式，合成模板仍是生成路径传入的 prompt；PPT plan 显式传 `pptLayoutPrompt` 时以它为模板，只有未显式传 prompt 时才回退到 `composerContent`。
7. count=1 时 root 节点同时持有 run summary 与 request trace；count>1 时 root 只持有 run summary，每个子节点持有自己的 request trace。`imageTask` 仅是活动句柄，请求终态仍保留。
8. 刷新后只允许根据已落盘的 `remoteTaskId` 续查；只有 `submitting` 而没有 task ID 时进入 `submission_unknown`，禁止自动重发。该阶段的“最多一次”边界限于单个活动标签页，不承诺多标签页互斥。
9. “再生成一稿”和“生成修改稿”都在 POST 前展示请求数及文生图 / 图生图构成，并且固定创建单请求 Run，不继承方案的初次多候选数量；候选修改不再把直连 `requestEdit` 的普通子节点混成 PPT 候选。
10. 每个 Request 保留最近 12 个状态事件；工作台按 batch 展示跨页 Run，Run 可展开全部 Request、task ID、预计/返回/完成数量和耗时，诊断复制会截断并脱敏认证信息。
11. Run/Request 节点、归属连线与未解决方案输入不接受普通上传替换、Agent mutation、撤销/重做或结构画布派生；工程仍有未解决请求时，画布列表和 PPT 列表都阻止删除。
12. Request 同时冻结 `channelId + baseUrl + apiFormat + model` 渠道身份（不持久化 API Key）。恢复原 task 前必须与当前配置严格匹配；渠道被删除、改地址/协议或移除模型时停在需处理状态，禁止回退到第一个渠道续查。
13. 远端已成功但交付图片过期或缺失时收敛为终态失败，同时持久化 `billingRisk`；之后的单页、批量、派生方案和按要求改图都要明示“可能重复计费”。

### 13.3 大提示词编辑器是本地事务

编辑器打开时复制当前有效提示词到 `editorDraft`，手工编辑、AI 改写和单槽撤销都只作用于本地状态。AI 请求由 `AbortController` + 单调递增 token 标识；手改提示词、修改改写指令、关闭、切页/切方案或卸载都会 abort 并使 token 失效，只有 token 仍匹配的响应可以覆盖草稿。取消不得写回节点；已有候选的锁定方案只能“保存为新方案”或“保存并生成新方案”。

本轮可靠性改造通过自动化可靠性测试 14/14、typecheck、build 与 localhost 本机假接口真实路径冒烟；冒烟期间仅向本机假接口发起 1 次文本分页请求，图片请求为 0，真实模型调用保持 0。完整人工验收项统一记录在“待测试”文档。

## 14. PPT Prompt Compiler：生成请求的唯一提示词事实源

PPT 页面生成不再把 outline、视觉方向、页面职责、方案排版和临时 prompt 分散拼接。两个向导入口都先形成 `DeckBrief + PageSpec[]`：`DeckBrief.styleContract` 是唯一 deck 级视觉事实源；`PageSpec.layoutRole` 表达封面、章节、内容、证据、对比或收尾职责；方案排版留在对应 target。不存在 page 级视觉 override。

`compilePptPromptSnapshot()` 是唯一编译入口。一个不可变 `CompilationSnapshot` 深拷贝并冻结完整 Contract、DeckBrief/PageSpec 版本、每个 `pageId + takeId` 的输入 target、最终提示词、来源与检查结果。Compiler 只消费 Contract 已冻结的 `direction`，不按 preset ID 查询最新目录，也忽略所有 `pptRole: "style"` 投影节点。六类页面职责分别进入最终提示词；空方向、损坏来源、未知 preset、缺失/未知职责以及 override 删除必要方向或职责都会形成阻断问题。

Contract 写入采用 DeckBrief 版本 CAS：真正变化才升版并清除全部页面确认，相同语义 no-op，过期版本失败；方案、候选和历史快照始终保留。页面职责或 `layoutIntent` 的变化只提升目标 PageSpec 版本并清除目标页确认。工作台快速生成和展开审阅消费同一份编译结果；直接编辑最终提示词保存为显式 override，override 与方案排版的人工确认都绑定完整精确文本。

生成计划把每个 Request 绑定到 `snapshotId + promptId + prompt`。Contract 参考图写入 `referenceSnapshots` 并按 `storageKey` 与普通图片去重，但不伪造 `inputRefs.nodeId`。可靠执行模块在 durable 写入和真正 POST 前重新读取工程，校验快照存在且未篡改、Request 提示词与快照完全相同、当前 Contract、PageSpec 与节点输入仍等于冻结值；参考 Blob 缺失、不可读或解析为空同样在 provider submit 前失败。已有远端 task 只恢复旧 task，不因当前 Contract 改变而重提。

候选确认与 ZIP/PPTX 导出还必须沿 `compilationSnapshotId` 或 `candidateEdit.baseNodeId` 回溯到唯一、同页同方案的 CompilationSnapshot。修改稿自身伪造的 snapshot ID 不参与判断；断链、循环、重复快照或跨页/方案借用会阻止确认，导出在读取图片 Blob 前终止。旧快照候选不要求与当前 Contract 相同，但必须由用户重新显式确认。

DeckBrief、PageSpec、编译快照、Run/Request 和图片引用仍保存在浏览器本地 localforage/IndexedDB，不代表云同步。自动化与 localhost 验证证明的是请求前的确定性编译、可见性和绑定；真实模型是否准确渲染全部文字与版式仍需单独的真实调用验收。

## 15. 精修动作与工作区状态

### 15.1 两个动作，一种追加语义

“再生成一稿”以当前方案原提示词新建 1 个候选，不包含修改要求；“按要求改图”以选中候选为基图，在同一界面编译整页要求与可选点位，再由“生成修改稿”新建 1 个候选。两者均强制 `plannedCount = 1`，不改写方案配置里的初次生成数，也不覆盖原候选或页面最终版。

### 15.2 修改血缘与可靠返回

修改 Request 除通用的 `pageId` / `takeId` / `runId` / `requestId` 外，还持久化 `candidateEdit`：基图 `baseNodeId`、规范化整页要求、带序号和相对坐标的点位快照、最终提示词。候选详情可据此说明修改稿来自哪张基图、按什么要求产生。完成通知同时携带 `runId`，返回工作台后选中该 Run 实际产生的候选，而不是仅定位页面或方案。

### 15.3 空间分配与会话内保持

中间栏保留原有三栏层级，只在上方方案说明区与下方候选区之间加入可上下拖动的分隔线；两区均有最小高度，矮视口则回退到原生紧凑滚动布局。分隔线支持键盘上下调整，且不引入额外高视觉重量。

工作台在会话内按 `pageId + takeId` 记住已选候选和候选区滚动位置，切页或切换方案后返回时会恢复。修改稿生成中保留基图并显示非阻塞运行态；用户可继续切页，完成后由可靠性通知带回真实候选。

## 16. 分层 PPTX POC 结论（技术实验，未产品化）

单页 POC 已证明“背景/主要视觉位图层 + 原生文字框”可以组成真正分层的 PPTX：Microsoft PowerPoint 能分别选择和移动背景、卡车、图标，编辑文字后保存重开也能保持。实验没有接入 PPT 工作台、没有修改正式工程数据，也没有新增导出入口；当前产品仍只支持每页整图、文字不可编辑的图片版 PPTX。

- 零调用装配和模型辅助装配均通过文件兼容性验证；每页包含 3 个独立图片对象和 7 个含文字的原生对象。
- 模型阶段只做了 3 次图片编辑、0 次重试；项目 API Key 直接费用为 $0，按 GPT Image 2 标准 API 高质量档保守折算上限为 $0.66。计价口径见 [OpenAI 图片生成定价](https://developers.openai.com/api/docs/pricing#image-generation-models)与[图片生成费用计算](https://developers.openai.com/api/docs/guides/image-generation#calculating-costs)。
- 自动文字识别和版面恢复没有被验证；模型拆出的卡车、图标、道路和城市轮廓出现明显重绘漂移，所以本实验不能支持“忠实还原”或“完全可编辑”的产品承诺。
- 裁决为**缩限继续研究、本轮不产品化**：图片版 PPTX 保持正式主线；若以后重启分层版，必须另立任务，先固定复杂代表页、数值质量阈值、自动文字/布局恢复方案和人工修复计时，再申请调用预算。
