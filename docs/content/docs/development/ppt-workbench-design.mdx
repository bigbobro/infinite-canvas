# PPT 工作台设计文档

> **文档性质**：本文档是 PPT 工作台模块的活文档（living doc），随功能迭代持续更新，反映的是当前实际状态，不是一次性设计快照。P0（材料 → 可编辑大纲 → 风格配置 → 建图落画布 → 面板批量生图（含首页锚定）→ 确认此页 → 图片 zip 打包下载）已上线（`890d305` + review 修复 `c0fcaf0`）。在此基础上新增了「已有规格，直接生图」模式——用户已在外部把每页要发给生图模型的完整提示词写好，工具只做逐字展开，不改写。新会话/新 agent 可从本文档直接理解当前架构，具体代码坐标见附录（合并上游后以符号名重新定位行号）。

## 1. 背景与目标

在现有生图能力之上，新增「PPT 工作台」模块，支撑以下业务流程：

1. **输入原始材料**：用户提供文字材料（Markdown 或整份文档），明确 PPT 核心内容。
2. **生成分页大纲（节点化）**：LLM 基于材料生成分页大纲（如 1–9 页），每页在画布上对应一个独立节点。
3. **风格控制（前置配置）**：生图前配置风格——参考图和/或风格描述（如「专业咨询报告风」）。
4. **批量生图**：结合大纲与风格，每页分别生成 16:9 页面图。
5. **多轮精修**：对每页生成结果做第二轮、第三轮乃至多轮调整。
6. **批量下载**：全部页面确认后，一次性打包下载整套图片。

## 2. 范围界定

- **交付物只有图片**：导出终点是按页序命名的图片 zip 包（`01_封面.png` … `09_总结.png`）。**明确不做 `.pptx` 生成**——那是另一套工作流，不在本模块范围内（用户拍板，2026-07-14）。
- **维持纯前端**：不引入任何后端/服务端持久化，数据全部走现有 localforage/IndexedDB 体系。
- **一个 deck = 一个专属画布工程**：PPT 结构数据住在 `CanvasProject` 内，导出/导入/WebDAV 同步免费获得，单一事实源。

## 3. 架构决策

### 3.1 选型：向导 + 画布 + 面板（混合架构）

对比过三条路：

- **方案 A：独立表单式工作台**（复刻 `/image` 三栏模式）——排除。步骤 2/5 与画布割裂，精修体验要在表单页重造一遍蒙版重绘/迭代链，重复建设。
- **方案 B：纯画布模板**——排除。没有流程引导、页状态追踪和批量下载落点，相比手动拼节点价值增量太小。
- **方案 C（采用）：混合架构**：
  - **步骤 1–3 用向导承载**：新路由 `/ppt`（进顶部导航，与生图/视频工作台并列），deck 列表 + 新建向导。向导第一步先选生成方式，两种模式共用后续落画布/面板/导出流程：
    - **「从材料拆大纲」**（默认，即 P0 原有流程）：材料输入 → LLM 生成分页大纲（**向导内可编辑：改标题/要点、增删页；确认后才落画布**）→ 风格配置（预设描述 + 参考图）。适合还没想清楚具体每页内容、需要工具帮忙规划的场景。
    - **「已有规格，直接生图」**：适合用户已经在外部（如与 AI 多轮对话）把每页要发给生图模型的完整提示词都写好的场景。工具的职责从「理解与规划」收窄为「定位与搬运」——LLM 只输出每页在原文中的起止行号，正文由代码从原文按行逐字切片，改写在结构上不可能发生，不需要事后校验。切片后对各页求公共前缀：命中（各页开头逐字相同）判定全局视觉风格已内嵌每页，风格步留空；未命中则从原文中未被任何页面占用的内容里提取候选文案预填风格步，交由用户整理确认（保真度要求低于页面正文本身）。
  - **确认后程序化建图**：构造完整 `CanvasProject`（节点 + 连线 + `ppt` 字段）经 `importProject` 入库，跳转 `/canvas/:id`。建图不需要画布页面在场。
  - **步骤 4–6 用画布内「PPT 面板」承载**（类似助手侧栏）：页状态总览、批量生成、单页重生成、确认此页、打包下载。
  - **步骤 5 精修完全复用画布既有能力**（自身参考再生成、蒙版局部重绘、裁剪、放大、多视角），面板只追踪每页当前确认的图。

### 3.2 执行层：PPT 编排伪装成一个「确定性 Agent」

`project.tsx` 是 3263 行的巨型组件，禁止深度侵入。项目已有声明式逃生门：**online/local 双 Agent 收敛到 `applyAgentOps`**（`add_node` / `connect_nodes` / `run_generation` 等 op）。PPT 面板通过 `useAgentStore` 的 `canvasContext.applyOps` 下发 `run_generation` ops，与 Agent 走同一条已被验证的管线，天然获得状态写回、请求取消、结果子节点自动创建。

这个选型有双重价值：

1. 复用测试过的生成管线，面板不碰生成逻辑一行代码；
2. 与上游的耦合面压到最薄——`project.tsx` 只占 1 个 import + 1 处 JSX 挂载（见第 7 节侵入点清单）。

顺带为 P2 铺路：将来加 `workbench_ppt_*` Agent 站点工具（本地 Codex/Claude 一句话生成整套 deck）几乎免费，落在「MCP/Agent 操作画布」这条差异化护城河上。

### 3.3 风格控制：走画布原生连线语义，不造新机制

画布上放一个「风格说明」文本节点 + N 张参考图图片节点，全部连入每页的生成配置节点。既有行为（`buildNodeGenerationContext`）保证：上游文本自动拼进 prompt、上游图片自动作为参考图走图生图（`requestEdit`）。风格对用户可见、可改，改完重生成即生效。

### 3.4 config 节点的 prompt 回写机制（易踩坑，改造 PPT 生成配置节点前务必先读）

`project.tsx` 的生成主函数（见附录「生成主函数」`handleGenerateNode`）在每轮生成结束后，会把**组装好的完整 prompt**（已含上游文本节点拼接进来的内容）回写进 config 节点自己的 `metadata.prompt` 字段。这意味着：若下一轮生成再从这个字段取值，上游文本会被二次拼接进去——实测会出现 1→2→3 倍的逐轮累积。

两种模式各用不同机制挡住这个回写：

- **「从材料拆大纲」模式**：面板每轮都硬传固定常量 `PPT_PAGE_PROMPT` 作为 op 的 `prompt`。`applyAgentOps` 的 `run_generation` 分流里，op 自带非空 `prompt` 会无条件优先于节点自身的 `prompt` 字段，被污染的字段因此从不会被读到。**这个看起来「冗余」的常量传参不能删——它是唯一的挡箭牌。**
- **「已有规格，直接生图」模式**：config 节点建图时额外写入 `metadata.composerContent: ""`。回写机制只碰 `prompt`、从不碰 `composerContent`；而 op 未显式传 `prompt` 时的取值顺序是「`composerContent` 优先，`prompt` 兜底，`??` 不对空字符串再兜底」，于是每轮都由这个空字符串胜出，面板因此可以停传常量。

两条已被实测排除、切勿重走的解法：

1. 面板改传节点当前的 `metadata.prompt`——这个字段本身就是被污染的那个，传它等于把污染值再传一遍，同样会累积；
2. 面板传空字符串、同时把取值判断改成「只要不是 `undefined` 就用 op 的值」——会把 `composerContent` 一并挡死，用户若用过配置合成器（`canvas-config-composer.tsx`）实测会下发 0 字。

后续任何人要新建/改造 PPT 的生成配置节点，先确认自己的写法有没有踩中这条回写路径。

### 3.5 标注改图（annotate）：画布通用能力，与蒙版重绘并存

图节点支持「标注改图」：在图上点位放编号红色气泡，逐点填写「改成什么」，可放多个、一次提交。模型收到的不是原图，而是烧了编号气泡的图片副本 + 按编号引用标记的指令（如「①标记所指的那个图标 → 改成火焰图标」……），原图节点保持不动；新图落在同一条精修链上（与源节点连线，既有连线语义自动纳入候选集）。

不走蒙版路线：蒙版是「刷一片区域 + 一段描述」，一个 mask 只能表达一个区域集配一个 prompt，表达不了「①改成 X、②改成 Y」这种多点各改各的——mask 不携带身份。两者并存、各管各的活：蒙版擅长大面积区域重绘，annotate 擅长多个离散对象各改各的。

这是画布的通用能力（不限图节点来源），任何图节点都可以用；需求源自 PPT 场景里图标级精修的诉求，但实现和挂载与 PPT 数据模型无关。

## 4. 数据模型

```ts
// CanvasProject（stores/canvas/use-canvas-store.ts）增加可选字段
ppt?: {
    sourceMaterial: string;           // 原始材料
    requirements: string;             // PPT 要求
    style: {
        description: string;          // 风格描述（预设或自定义）
        // 风格参考图。必须是「带 storageKey 属性的对象」数组：cleanup 的
        // collectImageStorageKeys 只收集对象上的 storageKey 字段，裸 string[]
        // 不会进 used-set，画布上的风格图片节点被删后 blob 会被清理误删。
        references: Array<{ storageKey: string }>;
    };
    pages: Array<{
        index: number;                // 页序（1 起）
        title: string;
        outline: string;              // "outline" 模式下是该页要点；"extract" 模式下是从原文逐字切片的该页完整内容
        visualHint: string;           // LLM 给的视觉建议；"extract" 模式下恒为空字符串
        confirmedNodeId?: string;     // 用户确认的最终图像节点 id

        // 该页全部线路（平行提示词）。一条线路 = 一对 outline/config 节点；新建 deck 只写这个，
        // 数组序即线路创建序（排序依据见 §6）。
        takes?: Array<{ anchorNodeId: string; configNodeId: string }>;
        /** @deprecated 用 takes；存量数据（单值）读时经 pageTakes() 视作 takes[0]，不写迁移脚本 */
        anchorNodeId?: string;
        /** @deprecated 用 takes；存量数据（单值）读时经 pageTakes() 视作 takes[0]，不写迁移脚本 */
        configNodeId?: string;
    }>;
    anchorConfirmed?: boolean;        // 首页锚定是否已完成
    mode?: "outline" | "extract";     // 建 deck 用的生成方式，缺省视为 "outline"（存量 deck 天然兼容）
};

// CanvasNodeMetadata（types/canvas.ts）增加的可选标记字段
pptPageIndex?: number;
pptRole?: "outline" | "style" | "page" | "source";   // "source" = 「已有规格，直接生图」模式下承载原始规格稿的节点
```

`pageTakes(page)`（`stores/canvas/use-canvas-store.ts`）是读取某页线路的唯一入口：`takes` 非空直接返回；否则若存量单值 `anchorNodeId`/`configNodeId` 都在，包成单元素数组返回；否则空数组。**读时归一，不写迁移脚本、不加版本号**——纯前端 localforage，存量 PPT 工程躺在用户浏览器里，写迁移有不可逆改坏用户数据的风险；新建的 deck 只写 `takes`，旧 deck 首次被「新增线路」时才自然升格出该字段。全仓所有读某页 outline/config 节点 id 的地方都经 `pageTakes()`，不直接读两个已废弃字段。

**防御性解析原则**：面板以 `ppt.pages` 为纲、以 metadata 标记做校验。用户在画布上删节点、改连线是自由的；面板发现某页结构缺失时提示「结构缺失」，不崩溃、不阻止用户操作画布。

`ppt.mode` 决定面板对该 deck 走哪套生成参数（详见 §3.4）；`pptRole: "source"` 节点只在「已有规格，直接生图」模式下出现，承载用户粘入的原始规格稿全文，**默认不与任何 config 节点连线**——它包含全部页面内容，连了会把全篇原文混进单页 prompt。画布原生支持用户自己手动拉线，工具不代为决定。

## 5. 画布工程结构（deck 的形状）

```
[风格说明 文本节点]──┐
[参考图1 图片节点]──┼──→ [第1页 生成配置] → (生成后出现) [第1页 页面图] → 精修链向右生长…
[参考图N 图片节点]──┘        ↑
[第1页大纲 文本节点]─────────┘
（每页一列，横向排布；第 2–N 页结构同上）
```

生成结果节点由既有管线自动创建并连线（`handleGenerateNode` 行为），精修每一轮自动形成可追溯的迭代链。

**一页多线路（take）**：一页可以有多条平行线路，每条线路是一对独立的 outline/config 节点（图中「第1页大纲」+「第1页 生成配置」只是其中一条）。面板「新增线路」追加一条新线路，**不销毁旧的**——原 outline/config 节点、已确认图、锚定状态都保持不动，旧线路仍是合法候选。新线路的 outline 节点初始内容复制自当前显示的那条线路（用户要的是「改」，不是「重写」）。

**P0 当前摆位**：新节点不落在当前页那一行的正下方。deck 建图的行距与「下一行」的间距相等，落在本行下方必与下一页那一行重合；因此当前实现取**整个 PPT 网格（所有带 `pptPageIndex` 的节点）的全局最低点**再加一个行距新起一行，x 沿用最新一条线路的列，并把画布视角带到新节点。P1 不改变这套底层节点摆位，但会用 §11 的页级 Focus 取代这段全局画布定位交互。

**P0 当前候选挑选**：每页的候选图（每条线路 config 下游全部成功生成的图，剪枝规则见 §6）按线路分组、组内保持创建序，铺成一条缩略图条，**候选数 ≥2 才显示**；点缩略图跳画布视角并选中该节点，hover 可直接「用这版 / 取消确认」。2round 实测已确认该窄面板方案无法支持辨认和比较，P1 按 §11 改为 Focus 内的 16:9 大图候选。

## 6. 关键设计难题与对策

1. **风格一致性（成败点）**——三层叠加：
   - 每页 prompt = deck 级统一风格前缀 + 该页大纲；
   - 相同参考图集连入每页；
   - **「首页锚定」两段式批量，默认开、可跳过**：先只生成第 1 页 → 用户精修确认 → 把确认图自动 `connect_nodes` 进第 2–N 页的配置节点 → 再批量生成。
2. **文字渲染准确性取决于模型能力**：当前使用的文字渲染强模型已经可以产出高信息密度的中文页面，不再把“中文小字天然不可用”作为产品前提。仍需按具体模型和页面实测，个别错字可用标注改图或蒙版重绘修正。交付物依然是按页图片，不提供可编辑 pptx；“图片型”描述的是交付格式，不再限制内容只能是低文字量、视觉冲击型页面。
3. **并发与失败处理**：现有机制无并发上限（生图工作台 count=15 也是全并发），9 页规模 MVP 直接沿用；**必须给单页失败重试**（面板每页独立重生成按钮），一页失败不毁整批。更长 deck 的并发队列（3–4 并发）留 P1。
4. **大纲质量**：向导内大纲**可编辑**（改标题/要点、增删页）后才落画布——大纲错了后面全错，这一步的人工把关是流程设计的一部分。
5. **候选归属靠剪枝 BFS，不靠裸遍历**：连线只是「谁指向谁」，边上没有语义字段——首页锚定会把第 1 页确认图连到第 2…N 页的 config 节点，若从第 1 页 config 做裸 BFS 收候选，会顺着这条连线把整个 deck 的图都吞进第 1 页（实测真实工程里第 1 页因此拿到 8 个候选、取末位得到第 6 页的图）。对策：BFS 遍历时若遇到带 `pptPageIndex` 且不等于当前页 index 的节点，立即停止（不收集、不入队）——`pptPageIndex` 只打在 outline/config 节点上，生成的图节点没有该字段，不会被误剪。批量生成（`count > 1`）的子节点同理排除：候选只收 batch 根节点（不带 `batchRootId`），子节点由画布原生 `setBatchPrimary` 在 batch 内部挑选，根节点始终持有当前挑中的那张——页级候选与 batch 内候选是两层，各管各的。

## 7. 上游同步策略（fork 迭代管理，保守评估制）

本仓库是 `basketikun/infinite-canvas` 的二开 fork（remote 已配好 `origin` + `upstream`）。

### 7.1 原则：上游发版只触发「评估」，不自动触发「合并」

每次上游发版，先做影响评估（读 release notes + diff 侵入点清单与依赖面），然后显式决策：

- **情况 1：上游新功能对我们无影响** → 仍要主动决策「并还是不并」。默认**不并**；只有明确想要该功能/修复时才并。不追新。
- **情况 2：上游改动影响到我们**（改了侵入点文件、`applyAgentOps` 管线、生成服务层等依赖面）→ 评估「改还是不改」：跟进适配，或暂缓合并冻结在当前基线。

每次评估**留决策记录**（追加到下表），合并后执行固定核对流程。

### maolao 异步生图（#9–#15，2026-07-17）：本清单第一次「改上游既有行为」

二开逻辑本体 100% 住在两个新文件：`web/src/services/api/maolao-image.ts`（协议：提交/轮询/取图/容错）、`web/src/pages/canvas/use-canvas-image-tasks.ts`（句柄落盘 + 重载续轮询）。上游文件只留分流与挂载。

但本次**无法做到全加法**，有三处改动了上游既有实现，必须显式登记：

| 位置 | 改了什么 | 为什么不能只做加法 |
|---|---|---|
| `normalizeApiFormat`（#9） | 三元判断 → 白名单 | 它会把任何非 `gemini` 值归一成 `openai`。不改，`maolao` 一存进 localStorage、下次读出即被吞掉，**功能静默失效且零报错** |

| `resetInterruptedGeneration`（#5） | 「loading 一律标 error」→ 三分支（持有未过期句柄则保持 loading 交由续轮询接管） | **这就是 R4 的本体。** 上游语义是「刷新即放弃」，异步渠道下等于放弃已付费且仍在运行的任务。已按 §7 要求先找上游既有语义替代——`composerContent` 那类「接管点」在此不存在，画布没有任何「任务可恢复」的表达 |

**已确认（用户拍板，「已有规格，直接生图」模式）**：

- **D1 本期范围**：只解决「用户已在外部把每页内容规划完整、写出完整生图提示词稿」这一类需求。「从零帮用户把模糊想法规划/改写成规格」的对话式扩展留待后续验证，不在本期做（见 D6）。
- **D2 改动范围的澄清**：不是「只能改向导」。向导第一步的模式选择只是**入口**改动；只要能让这个新生成方式正确工作，下游的建图逻辑（`deck-builder.ts`）、画布面板（`canvas-ppt-panel.tsx`）等文件都可以按需调整——§3.4 的 `prompt` 回写挡板正是靠面板改动解决的。
- **D4 模式选择方式**：不做「自动识别输入类型 + 告知判断结果 + 允许用户推翻」，改为向导上**显式**选择两种模式之一，默认落「从材料拆大纲」（老流程）。理由：两种误判代价不对称——把一份已经写好的完整规格稿误判成需要重新规划，会导致内容被复述改写，等于线上复现了本期要修的问题；而显式选择的实现更小、误判率为零。
- **D5 规格节点默认不连线**：用户粘入的原始规格稿在画布上单独建一个节点保存全文（单一事实源），但**默认不连**任何生成配置节点——它包含全篇内容，连了会把全篇原文混进单页 prompt。画布原生支持用户自己手动拉线，工具不代为决定。
- **D6 对话式扩充暂不做**：帮助只有模糊概念、还没规划好每页内容的用户通过多轮对话把内容「聊」出来，是有价值的方向，但作为独立能力**是否可靠尚未验证**，本期不做，留作后续任务。

**已确认（用户拍板，PPT 多版本模式）**：

- **D7 候选归属靠剪枝 BFS，不靠节点自带标记**：备选方案是让生成的图节点继承 `pptPageIndex`（归属可直接查字段，不用遍历），但要往 `handleGenerateNode`（`project.tsx` 里改动最活跃的函数）伸手——不值得为「标记自带 vs 算出来」赌上游合并冲突。剪枝 BFS 反正每次渲染都要跑，查表是顺带的。
- **D8 一页多线路用 `takes[]`，不做画布自动视觉分组**：归属靠 D7 的剪枝 BFS 保证正确，不需要额外的视觉分组机制兜底；画布原生 Group 是几何包含（拖进框即属于该组，拖出即脱离），生成的图落在算好的坐标上未必落进组框，不能拿来做归属判定。P1 仍不整理底层画布，但会按 §11 在页级 Focus 中聚合显示，不再要求用户靠拖节点完成精修定位。
- **新线路初始内容 = 复制原文**：新增线路时 outline 节点内容复制自当前线路，而不是留空或让 AI 改写——用户要的是「改」，不是「重写」，给一份副本直接编辑最省事。
- **候选排序 = 按线路分组 + 组内节点数组序，不引入 `metadata.seq`**：`CanvasNodeMetadata` 没有时间戳字段，「最新」原本就是「`project.nodes` 数组末位」这个隐含约定；铺开候选后按线路分组展示（这版是哪条线路出的一目了然）已经足够说清顺序，不必新增字段——数组序会被 undo/import 扰动的风险先接受，真出问题再加序号。

**已确认（用户拍板，标注改图 annotate）**：

- **技术路线选 B（烧标记单图），不选 A（坐标转文字）/ C（双图）/ 蒙版**：A 在密集目标（图标挨在一起）时纯文字描述指认不清，实测还丢了保真细节（圆环）；C 成本翻倍，且唯一优势（结构上防标记泄漏）经实测非必要——B 的标记从未泄漏；蒙版的 mask 不携带身份，表达不了「多点各改各的」（一个 mask 只能配一个区域集 + 一个 prompt）。7 次真实 API 调用全部验证过：仅给位置、不给物体名即可准确命中，不需要额外的视觉识别步骤。
- **挂载方案选 B：自持状态的宿主组件，不照抄蒙版对话框的挂载范式**：蒙版对话框在 `project.tsx` 里有 8 个有状态触点，是上游自己的代码风格；二开照抄等于把这个巨型组件从 2 触点炸到 9，「复用其挂载点」也不成立，等于深度侵入另一个上游文件。改为状态住二开自有 store（`useAnnotateStore`），节点数据经 `canvasContext.snapshot` 读，生成路径自走 `requestEdit` + `applyOps` 建子节点/写状态。该方案在 `project.tsx` 落下 4 处触点，全部为加法（无一改动上游既有行为或函数体），但超出本清单原定「1 import + 1 挂载」上限，已获用户批准破例（2026-07-15），条件是全部为加法、且每处带 `[二开]` 标记（§7.4 #5 已按此登记）。
- **否决工具面板版本常量 v6 → v7 升级**：曾考虑借上游「版本号递增」的既有惯例，让存过工具面板配置的浏览器自动看到新工具。否决理由：① 改的是上游既有行，且升版会重置用户已保存的工具自定义；② 当下收益为零——本机没有存量配置，annotate 按默认工具列表已经可见。已知边角记档不改码：存过 `canvas-image-quick-tools-v6` 配置的浏览器需要用户自己在工具面板里勾选，才能看到「标注改图」。

**留待实现期细化**：风格预设的具体文案清单、并发队列阈值、16:9 的具体 size 档位映射。

## 11. P1 精修与交付交互评估（2round 实测）

本轮对智能纪要、原始录音和本地真实工程完成交叉核对。长 prompt 导入、分页和生成已经满足当前使用；阻塞集中在生成后的精修与交付：新增线路落在全局画布底部、48px 方形候选看不清、标注视窗太小、6/6 页确认后仍直接下载而没有最终检视。

交互主对象调整为 `页 → 线路 → 版本`，推荐新增 PPT 私有的**页级 Focus 工作区**：右侧面板只做页导航；工作区聚合当前页全部线路和版本，以 16:9 大图比较、标注和确认；底层继续复用现有节点与连线，不使用上游原生 Group，不永久重排全局画布。

最终交付必须拆成独立阶段：所有页都显式确认后，按第 1–N 页大图检视，再允许打包。最终导出只接受 `confirmedNodeId`，确认节点、`storageKey` 或本地 Blob 任一缺失都整体失败；不再 fallback 最新候选，也不再静默跳页。

该方案已在独立分支按上述边界实现：新增页级投影、Focus 工作区和最终检视，放大现有 annotate 私有弹窗，并将 ZIP 导出改为显式确认、逐页完整校验。实现只涉及 `web/src/lib/ppt/`、`canvas-ppt-*` 和现有 annotate 私有文件，没有新增 metadata、修改生成主流程或增加上游侵入点。

零付费 UI 验收已覆盖多线路/多版本聚合、标注大视图、缺页阻断和 6/6 就绪状态；真实新增线路与严格 ZIP 内容也已完成用户实测，前置 Focus 任务已经归档。上游合并继续作为独立后续任务，不与本次交互实现混在同一代码任务中。

## 12. 当前迭代实施：工作台主界面化与方案分支提示词

P1 同步实测带来了一个比“整理画布”更根本的结论：对 PPT 工程而言，用户的主对象不是节点和连线，而是 `页面 → 方案分支 → 候选稿`。现有页级 Focus 已经比全局画布更接近真实工作方式，因此后续不应继续让 PPT 工作台附属于画布；**PPT 工作台应成为默认主界面，结构画布退为按需进入的高级视图**。

这里统一把当前界面的“线路”定义为**方案分支**。新建分支的业务含义是“保留当前方案，同时修改提示词探索另一种方案”，所以系统必须先复制当前分支提示词，给用户编辑空间，再由用户主动生成。分支尚无候选稿时可以继续编辑；一旦已有候选稿，原分支提示词保持稳定，后续调整通过“基于此方案调整”派生新分支，避免同一分支的候选实际来自不同提示词。页面的最终确认仍属于页面，指向任一分支中的一个候选稿。

架构上暂不直接建立独立 `/ppt/:id` 运行时。当前 `project.tsx` 和 `canvasContext` 仍承载节点状态、生成、取消、持久化与 MAOLAO 异步续跑；复制这套能力风险过高，在合并后的新上游结构确定前抽取运行时也会放大冲突。近期采用**同一运行时、两个界面**：PPT 工程默认显示覆盖画布内容区的全尺寸 Workbench Surface，画布保持挂载；用户需要理解节点结构或做高级编辑时切换到“结构画布”。非 PPT 工程不改变。

分期顺序如下：

1. **先补功能**：在现有页级工作区加入分支提示词草稿、编辑和“基于此方案调整”；创建分支与发起生成拆开，不自动消耗 API。
2. **再升主界面**：把 Modal 形态的 Focus 提升为 PPT 工程默认的全尺寸工作台，补工作台/结构画布切换和双向定位。
3. **远期再拆运行时**：完成当前实测和上游合并后，只有在独立路由或后台渲染确有收益时，才另评估共享 `ProjectRuntime`；不得为 PPT 复制生成和任务恢复逻辑。

阶段 1、2 已在当前任务中实现，并通过零 API 浏览器复核和用户体验验收：PPT 工程默认显示全尺寸工作台；新建方案分支先编辑提示词，创建与生成分离；已有候选的分支通过“基于此方案调整”派生；结构画布保持挂载，并可在切换时定位当前候选或分支节点。浏览器复核还覆盖了未保存提示词拦截、隐藏画布键盘隔离、标注大视图、最终检视、刷新持久化、跨工程状态，以及返回结构画布后的真实拖动、缩放、节点选中与点击。实现继续复用 `CanvasProject`、`canvasContext` 和原生成管线，没有抽取第二套运行时，也没有发起真实模型调用；MAOLAO 生成中刷新续跑仍留待后续上游合并任务按真实路径验收。

当前只在 PPT 工作台流程内约束“已有候选后提示词只读”。结构画布作为高级入口，仍允许用户直接编辑底层 outline 文本节点；这是有意保留的高级编辑逃生口，不是全局节点硬锁。画布多线路自动分组、自动整理、永久重排和独立运行时继续后置。详细决策见 `CONTEXT.md`、`docs/adr/0001-ppt-workbench-primary-surface.md` 和 `.trellis/tasks/07-17-ppt-workbench-primary-surface/`。
