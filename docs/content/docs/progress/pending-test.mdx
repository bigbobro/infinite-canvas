## 下载图片版 PPT

- **交付边界**：最终检视新增“下载图片版 PPT”，并在入口下方明确说明“每页为整张图片，文字不可编辑”；下载的 `.pptx` 每页只铺一张整图，不应误解为原生字体或对象可编辑的 PPT。
- **最终确认与本地图片**：PPTX 和页面图片 ZIP 都只按页序使用每页显式确认的最终稿，从 `storageKey` 读取浏览器本地 Blob，不用最新候选或第一张候选兜底。任一页未确认、Blob 缺失或读取失败时，应点名具体页并阻止两种严格导出。
- **PPTX 与 ZIP 分流**：统一页面比例的 PPTX 应按原比例全幅铺满、不拉伸不裁切；页面比例混合时点名异常页并只禁用 PPTX 入口，“下载页面图片（ZIP）”仍可正常使用。
- **文件与零调用**：核对 PPTX 页数、页序、最终稿和安全化工程文件名，并用 Microsoft PowerPoint 及一种兼容查看器打开；整个下载过程不请求文本、文生图或图生图模型，真实调用次数必须为 0。

## PPT 精修与工作区体验

- **修改入口**：从候选稿进入“按要求改图”，分别验证仅整页要求、仅一个或多个点位、整页要求 + 多点位三种组合；清除文字和清除点位互不影响，提交前的修改清单、最终提示词和确认内容完全一致。零点位时不得出现红色标记或“所指对象”等话术。
- **固定单张**：先把方案生成数设为 3，再分别执行“再生成一稿”和“生成修改稿”；两者都应明确为 1 张 / 1 次调用，只追加一个候选，方案原有的 3 张配置不变。“再生成一稿”不带修改要求。
- **原稿与定稿保护**：两种精修动作都不覆盖基图、不改写已确认最终版；修改稿生成中仍能查看基图，界面明确显示“正在基于此稿生成修改稿”，并且可切换页面继续工作。
- **候选追溯与定位**：修改稿刷新后仍能查到所属 `pageId` / `takeId`、基图、整页要求、点位快照、最终提示词、`runId` 和 `requestId`；点击完成通知时按 `runId` 选中该轮真实候选，旧 Run 晚完成也不得定位到新 Run 的结果。
- **上下区域调节**：在中间栏上方方案说明区与下方候选区之间上下拖动分隔线，确认两区都保留最小可操作高度；切换页面或方案再返回时，已选候选和该方案的滚动位置保持。
- **尺寸、主题与回归**：在 1512×700、1512×785 与浅色 / 深色主题下确认主图、候选、按钮和分隔线无溢出；同时回归新建方案的大提示词编辑区、最终检视的页码 / 方案 / 候选序号 / 确认状态，以及返回结构画布后的拖动、缩放、选中和点击。
- **验证边界**：本节可使用静态图片和假任务完成自动化、typecheck、build 与 localhost 验收；未经单独确认调用预算时，文本、文生图和图生图真实模型调用均必须为 0。

## PPT 视觉方向 Contract V1

- **旧本地工程列表防崩溃**：在 localhost 保留早期创建、缺少 `pageId` 或 `takes` 的 PPT 工程，刷新“我的 PPT”应继续显示工程卡片，不再进入 React Router 错误页；异常工程只在列表读模型中安全降级，不自动迁移或改写本地数据。
- **唯一视觉事实源**：structured 新建 PPT 的“视觉方向”仍保持为第三步，默认选中“清晰专业”，不填写额外文字也能建图；“视觉叙事”“品牌优先”、自定义方向和参考图都应落到 `DeckBrief.styleContract`，不再创建或读取可编辑风格节点。verbatim 只有经用户明确确认的整套说明，不自动创建默认 Contract。
- **作用域清楚**：工作台头部可修改整套 deck 的视觉方向；当前页高级区可选择封面、章节、内容、证据、对比或收尾职责；方案排版仍只作用于当前方案。材料页的 PPT 要求只接受内容、页数和禁止项，页面建议只接受构图或素材；其中出现视觉风格时必须提示移到 Contract，最终提示词不得形成第二视觉来源。
- **失效但不删历史**：真正改变视觉方向后，DeckBrief 升版并清除全部页面确认，历史方案、候选和编译快照保留；只改变页面职责或页面布局时，仅目标 PageSpec 升版并清除目标页确认。重复保存相同内容不升版，过期编辑必须明确失败。
- **参考图与付费门禁**：Contract 参考图与普通输入图按 `storageKey` 去重并进入冻结请求，但不伪装成画布节点；Blob 缺失、不可读或解析为空时应在提交前失败。旧计划在 Contract/PageSpec 变化后不得 POST，已有远端 task 仍只恢复原 task。
- **候选血缘与导出**：普通候选和多级修改稿都必须回溯到唯一、同页同方案的 CompilationSnapshot；断链、循环、重复快照、伪造或跨页/方案借用时，确认及 ZIP/PPTX 导出都应阻止且不读取图片 Blob。只更换视觉 Contract 时，旧快照候选允许由用户重新显式确认；整套内容定位、PageSpec 或原始来源已漂移的旧候选不得重新确认。
- **验证边界**：本节只用静态数据、假 Provider 与 localhost 验收，文本、文生图和图生图真实模型调用均为 0；不据此承诺真实模型的视觉质量。

## PPT 内容方案与 Prompt Compiler

- **三步正向流程**：用“从材料拆大纲”进入新建向导，确认步骤名依次为“材料与目标 / 内容方案 / 视觉方向”；内容方案未通过审核时不能进入视觉方向，进入第三步后只调整整套视觉规则，不再改写已确认正文。
- **视觉线索交接**：内容检查识别到的整套审美描述不应阻断内容确认；进入“视觉方向”后，线索区默认展开、原文可见且需用户显式点击“采用线索”后才进入整套自定义方向。
- **ContentBlock 与来源**：检查每页包含唯一标题、唯一核心信息及其他正文、列表、表格、图表数据或占位块；材料、要求、用户回答和已确认假设分别形成可追溯 `sourceRef`。伪造行号、无关来源、原材料变更后失效的切片、无来源结论、排版要求夹带的新事实和越权视觉映射必须形成 blocking 问题，不能悄悄进入 PageSpec 或生成请求。
- **缺口、审核与事务编辑**：缺材料、缺证据、无来源结论或歧义要显示为具体信息缺口；标题、核心信息和整套定位不能用省略或占位绕过。安全修复、编辑、删页、合并、排序及单页重生成都绑定 draft revision，过期结果原子失败；只有无 blocking 问题的方案才能确认成 `approved` PageSpec。
- **两类 canonical 规格**：材料规划工程必须落为 `compilePolicy: "structured" + DeckBrief + PageSpec[]`；“已有规格，直接生图”必须落为 `compilePolicy: "verbatim" + VerbatimSpec[] + 可选 confirmedGlobalSpec`。两类分支不互转、不从节点正文或旧 `mode` 字段兜底；逐字正文的节点投影被改写时，最终提示词仍只来自 VerbatimSpec。
- **统一 selector 与 fail-closed**：列表、工作台、生成和导出统一用页面 descriptor 校验页面账本、连续页序、页面/规格一一对应，以及 structured 的标题/核心信息或 verbatim 的标题/逐字正文。损坏时显示“内容规格待修复”，生成计划应为 0 runs / 0 requests；跨 policy、缺快照、快照篡改、canonical 内容漂移、逐字模式节点 override 或额外布局都必须在 provider POST 前失败。
- **最终提示词与血缘**：structured 快照冻结 DeckBrief、目标 PageSpec、target、提示词、来源和检查结果；verbatim 快照冻结目标 VerbatimSpec、已确认整套说明和 target。每个 Request 绑定同 policy 的不可变 CompilationSnapshot，候选确认及 ZIP/PPTX 导出拒绝跨 policy 或无法确定性重建的血缘。
- **结构画布保护**：PPT 内容规格、视觉方向、生成配置、候选和 Run/Request 归属只由工作台修改；结构画布仍可拖动、缩放、选中、查看和下载，但不能经通用编辑、生成、重试、替换、删除、连接、撤销、Agent 或图像派生工具绕过工作台。
- **验证边界**：本轮只使用静态数据与假 Provider 验证 canonical、审核和请求前门禁；文本、文生图和图生图真实模型调用均为 0，因此仍需单独验证真实文本模型的内容规划质量及真实生图模型的文字、版式质量。

## PPT 生成可靠性

- **稳定身份**：新建 PPT 后新增、复制、切换和删除方案；页面排序与节点引用变化时，页/方案选择、确认图和候选归属仍保持正确，不再用页序或配置节点 ID 充当业务身份。
- **保存后再请求**：批量与单页生成都先写入结构和固定请求槽，经 localforage 读回并确认 `submitting` 锁后才发请求；落盘或读回失败时不得产生 POST。确认后到 POST 前替换共享参考图时，请求仍使用确认瞬间冻结的图片快照。
- **多图请求槽**：分别验证 count=1 与 count=3。count=1 的同一节点同时持运行摘要和请求 trace；count=3 为一个 root + 三个固定子槽，每槽最多一次 POST，成功后在原槽变成候选；其他槽仍更新进度时，用户已选择的主图和展开状态不能被旧持久快照覆盖。
- **刷新恢复**：异步任务处于 submitted/running、远端成功但尚未保存、结果保存失败三个阶段时刷新，只继续查询原 task ID，不新增 POST、不重复候选；通用画布恢复 hook 不得同时接管 PPT 请求槽。
- **原渠道恢复**：请求槽冻结 channel ID、地址、协议和模型，但不保存 API Key。任务运行后删除渠道、改地址/协议或移除模型，重新获取应停在需处理状态并要求恢复原配置，不得默认切到其他渠道。
- **未知提交**：在 durable `submitting` 后、task ID 落盘前中断，重开后显示“提交结果未知”且不自动重提；可标记放弃。此时再次生成必须明确提示可能重复计费，并创建全新的 batch/run/request。
- **重新生成预算**：已有历史 Run/候选时，普通重新生成再次显示新请求数与文生图/图生图构成；未知请求标记放弃后，单页和批量入口仍保留重复计费警告，不得因状态已变成 `abandoned` 绕过确认。
- **按要求改图**：PPT 候选生成修改稿前确认 1 次图生图，并创建新的 batch/run/request；刷新恢复、终态候选与诊断均走同一模块，不再产生无台账子节点。
- **终态与通知**：完成、部分完成、失败和需要处理的应用内提示每个 run/终态只出现一次，点击能回到对应页；刷新后不重复提示。工作台“生成记录”按 batch 展示跨页每个 Run，并可展开 Request、task ID、预计/返回/完成数量和耗时。错误区可复制不含提示词、图片正文、API Key 的结构化诊断。
- **错误分类**：无 task ID 的 5xx/408/断网按提交结果未知处理；已有 task ID 的临时轮询或取图错误保持可重新获取；明确远端终态、可确认未执行的 4xx 拒绝或已知 0 POST 的本地准备失败显示中性的“已失败”，并保留具体原因。远端已成功但图片过期或缺失时，还要持久化重复计费风险，之后的单页、批量、派生方案和按要求改图均必须再次警告。
- **台账保护与回归边界**：普通画布与 Agent 的非 PPT `run_generation` 保持原行为；PPT 归属节点不能通过结构画布生成/重试、上传替换、输入改写、删连线、撤销/重做或普通图片派生绕过请求槽。未解决请求存在时，从“我的画布”“我的 PPT”和工程内都不能删除工程；返回结构画布后仍要实际拖动、缩放、选中和点击，确认没有不可见弹层拦截。
- **验证边界**：本轮自动化可靠性测试 14/14、typecheck、build 与 localhost 本机假接口真实路径冒烟均通过；localhost 仅向本机假接口发起 1 次文本分页请求，图片请求为 0。真实模型调用保持 0；建议人工真实复测上限仍为文本 0、文生图 2、图生图 1，执行前需再次确认模型、单价与费用。
