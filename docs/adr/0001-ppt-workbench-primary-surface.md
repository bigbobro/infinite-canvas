# ADR 0001: PPT 工作台作为 PPT 工程默认主界面

## Status

Accepted

## Context

当前 PPT 工程运行在 `/canvas/:id`，PPT 面板和页级 Focus 叠加在通用画布上。实测表明，用户的主要任务是按页调整提示词、比较方案、标注、确认并交付；全局节点画布对这条主流程的价值较低，反而增加了线路定位和版本理解成本。

与此同时，现有画布页面持有生成、持久化、任务取消和 MAOLAO 异步任务续跑等关键运行时。现在直接建立独立 PPT 工程页面，会迫使项目先拆分巨型运行时，或者复制高风险逻辑。

## Decision

- PPT 工程默认显示全尺寸 PPT 工作台。
- 结构画布保留为同一工程的高级视图，可从工作台明确切换进入。
- 近期在现有 Canvas 工程运行时内提供双界面；工作台显示时画布可被覆盖，但保持挂载。
- 工作台和结构画布共享同一个 `CanvasProject`、生成管线、持久化与异步任务状态，不建立平行实现。
- 独立 `/ppt/:id` 路由和 `ProjectRuntime` 抽取推迟到上游合并后另行评估。

## Considered Options

### Canvas 优先，Focus 作为弹窗

改动最小，但继续把次要的结构视图当作主产品界面，无法解决用户心智和操作重心错位。

### 同一运行时的工作台/结构画布双界面

能直接复用现有 Focus、节点数据和生成运行时，近期风险最低。代价是隐藏画布仍有后台渲染、焦点和快捷键隔离成本。

### 独立 PPT 路由，共享抽取后的 ProjectRuntime

长期边界最清晰，但当前必须先重构上游活跃的 `project.tsx`，并重新验证生成与任务恢复。收益尚不足以抵消近期合并风险。

## Consequences

- 后续 PPT 功能首先围绕工作台设计，结构画布不再承担主要精修流程。
- “方案分支提示词”成为一等交互，新增分支先编辑、后生成。
- 近期不做画布自动分组和永久重排。
- 实施工作台覆盖层时，需要验证隐藏画布的焦点、快捷键和渲染成本。
- 将来若抽取运行时，必须保持单一事实源和单一生成管线。
